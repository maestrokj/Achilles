# Open Questions

Вопросы на проработку. Обсудить по мере продвижения реализации.

## Multi-tenancy

**Суть:** нужна ли поддержка нескольких организаций в одном инстансе?

**Контекст:** сейчас Achilles — self-hosted (одна компания = свой деплой). Multi-tenancy понадобится только если появится SaaS-версия. Затрагивает все слои: PostgreSQL, pgvector, Redis, API.

**Решение на сейчас:** не закладывать. Держать чистую архитектуру (repository pattern, не хардкодить глобальность конфигов), чтобы миграция к MT была посильной когда/если понадобится.

## Email — асинхронные bounce'ы

**Суть:** модуль Email ловит ошибки доставки только синхронно — по ответу SMTP в момент `send` (4xx → ретрай, 5xx → «не доставлено»). Письмо, принятое релеем и отскочившее позже асинхронным bounce / DSN-сообщением, не отслеживается: входящего ящика и разбора bounce нет.

**Решение на сейчас:** вне v1. Для self-hosted SMTP это потребовало бы отдельного входящего конвейера (IMAP / каталог bounce + парсинг DSN) — несоразмерно для транзакционной отправки. Достаточно синхронной таксономии ошибок и ретраев очереди. Вернуться, если появятся жалобы на «молча потерянные» письма либо провайдер с webhook-нотификациями о bounce.

## Query Engine — масштаб истории диалога при длинном разговоре

**Суть:** на длинной беседе история диалога и найденный контекст конкурируют за единый бюджет контекстного окна модели; стратегия v2-уровня (running summary старых ходов или гибрид поверх token-window) не задана. Единая картина и v1-решение оформлены — `query-engine/_workzone/conversation.html#context-budget`.

**Контекст:** история живёт в `messages` (append-only) и растёт неограниченно. Окно — общий ресурс хода: его делят system, история, текущее сообщение, найденный контекст (`augment` усекает его под остаток) и резерв под ответ. Усечение ≠ потеря: дословная переписка остаётся в `messages`, факты перевыбираются свежим retrieval из KS — окно лишь рабочий буфер вызова. Открытым остаётся только v2: token-window рвёт дальний контекст, суммаризация держит его ценой лишнего инференса и риска потери редкой детали.

**Решение на сейчас:** v1 — token-window под общим бюджетом окна (дом — `conversation.html#context-budget`): история усекается с конца по токенам, augment кладёт фрагменты в остаток. Running summary — v2-форк, вводить, когда замеры на реально длинных диалогах покажут потерю контекста. Модель данных не меняется: окно — параметр чтения `messages`, а сводка (если понадобится) ляжет в `conversations.meta` или отдельный столбец без переделки.

## Query Engine — удаление и retention диалогов

**Суть:** в модели данных задан только каскад при удалении пользователя (`conversations` / `messages` ON DELETE CASCADE). Пользовательское управление беседами (список, переименование, удаление), политика retention (срок хранения, автоочистка) и экран списка бесед не спроектированы.

**Контекст:** беседа — персональный тред пользователя; привычный UI даёт список, переименование и удаление. Дом данных — Query Engine (`conversations`, `messages`, `retrieval_trace`), но экраны управления принадлежат оболочке Web App (и нативным тредам Slack / MCP), которая ещё не спроектирована. Открыто и то, нужен ли срок хранения (корпоративная политика, «право на забвение») или беседы живут бессрочно до ручного удаления.

**Решение на сейчас:** вне текущего объёма. v1 покрывает создание и продолжение беседы; список / переименование / удаление и retention решить при проектировании Web App, где живёт экран бесед. Удаление беседы ляжет на уже заложенный каскад (`messages`, `retrieval_trace` уходят за `conversations`) — модель данных готова, нужны лишь эндпоинт и экран.

## Query Engine — перенос сигнала спроса при мёрже сущностей KS

**Суть:** `access_counter.entity_ref` и ссылки `entity_id` в `retrieval_trace.citations` указывают на id сущности Knowledge Store. Когда Curation Pass сливает дубли (`knowledge-store/_workzone/lifecycle.html#staleness`, шаг «Мёрж нод» — это v1), id дубля перестаёт быть каноническим: `entities.trust_score` и ранжирование идут по сущности-наследнику. Накопленный на старом id спрос (`hits` / `last_accessed_at`) на наследника не переносится, а свежие обращения по старому id продолжают капать мимо канона. Кто реконсилирует внешние ссылки спроса при мёрже — не задано.

**Контекст:** строка-дубль при мёрже физически не удаляется (помечается `duplicate_of`), поэтому висячего указателя / FK-нарушения нет — это кросс-модульная ссылка без инфорсмента. Деградация мягкая: пока строки `access_counter` под id канона нет, staleness берёт нейтральный множитель (`lifecycle.html` — деградация по недостающему сигналу). Но история востребованности слитой сущности молча теряется, и сигнал расходится с каноном до тех пор, пока на наследника не накопится свежий спрос. Дом счётчика — Query Engine; мёрж — у Knowledge Store, поэтому перенос затрагивает обе стороны границы (`query-engine/_workzone/retrieval.html#access-signal`).

**Решение на сейчас:** вне v1, расхождение терпимо (мягкая деградация, не катастрофа). Перенос решить вместе с deep entity resolution в Curation Pass: при мёрже сворачивать `access_counter` дубля в строку наследника (сумма `hits`, максимум `last_accessed_at`) — естественная часть шага «перенос рёбер», дом механики — Knowledge Store, дом данных — Query Engine. Модель данных не меняется: перенос — операция над уже существующими строками по `entity_ref`.

## Knowledge Store — денормализация ACL в массив для vector pre-filter

**Суть:** ускорить ACL-фильтр в горячем текстовом поиске (vector + lexical), денормализовав гранты доступа в массив групп на сущности/фрагменте (`acl_groups int[]` + GIN), вместо многотабличного резолв-джойна на каждом запросе. Затрагивает `knowledge-store/_workzone/acl-identity.html` и `data-model.html#chunks`.

**Контекст:** в v1 ACL — живой JOIN (`users → identity → source_principal → group_membership → source_group → entity_acl`), и он же единственный фильтр текстового поиска, приходящий через JOIN на `entities`/`entity_acl`. pgvector (и GIN) применяют фильтр **после** скана индекса, а `hnsw.iterative_scan` компенсирует недобор только для условий на самой `chunks` — JOIN-предикат он не докручивает. Денормализованный `acl_groups && :my_groups` стал бы локальным квалом: iterative_scan смог бы его добирать, а резолв-джойн на запросе исчез бы. Это **не** pre-filter внутри обхода графа (такого в pgvector нет) — выигрыш в том, что ACL перестаёт быть некомпенсируемым JOIN-квалом.

**Решение на сейчас:** не закладывать в v1. Цена — write-amplification: смена членства в группе или отзыв гранта переписывает массивы у всех затронутых строк, права становятся eventually consistent на уровне денормализации. Сырой реляционный ACL (пять таблиц, резолв-джойн) проще и честнее как source of truth. Composite-индексы `entity_acl` под направление резолва (`(source_group_id, entity_id)`, `(source_principal_id, entity_id)` — заложены в `acl-identity.html`) закрывают значимую часть выигрыша без денормализации. Вернуться при доказанном масштабе, когда замеры покажут, что JOIN-фильтр роняет recall/латентность текстового поиска.

## Query Engine — надёжность tool-calling у слабых моделей

**Суть:** в новой концепции (`query-engine/_workzone/conversation.html#route`) chat-модель сама решает, звать ли инструмент `search_knowledge`, и формулирует самостоятельный запрос как его аргумент. Это опирается на способность модели к tool-calling. Achilles даёт компании подключать **любую** chat-модель, в том числе слабую или локальную, которая ненадёжно решает звать инструмент или плохо формулирует запрос.

**Контекст:** опасный режим отказа — ложное «это болтовня»: модель не позвала поиск на вопрос к знаниям и ответила из общих знаний, выдумав корпоративный факт. Смещение в сторону поиска (политика промпта) снижает риск, но не убирает у моделей, плохо держащих tool-calling. Возможные фолбэки: форсированный retrieval (всегда искать) для помеченных «слабых» моделей, либо дешёвый служебный роутер-классификатор перед моделью.

**Решение на сейчас:** v1 предполагает tool-способные chat-модели; смещение в сторону поиска задаётся промптом. Фолбэк для слабых моделей (форсированный поиск или дешёвый роутер) — v2, ввести при реальной потребности, когда в каталог попадут модели со слабым tool-calling. Модель данных не меняется: решение о поиске — поведение прохода, не схема.

## Query Engine / Admin — егресс контекста во внешний LLM-провайдера

**Суть:** при grounding-ответе упакованный контекст (найденные корпоративные фрагменты) уходит выбранной chat-модели; если модель облачная, фрагменты покидают периметр. Allow-list моделей (`admin-panel/_wireframes/ai-models.html#model-assignment`) сужает выбор по стоимости / доступности, но не по чувствительности данных: оси «класс данных → допустимая модель» (облако запрещено для закрытого класса) и редакции / минимизации контекста перед отправкой нет. Риск назван в `query-engine/_workzone/protection.html#egress`.

**Контекст:** смежно с «AI-модели — область видимости пользовательского списка», но это другой разрез — комплаенс / data-residency, а не «кто какие модели видит». Технически локальные провайдеры (Ollama / vLLM) уже поддержаны (`ai_providers.kind = local`), то есть «гонять чувствительное локально» возможно как выбор инфраструктуры, но не как гарантия. Сама классификация чувствительности данных в Knowledge Store ещё не заложена — без неё ось «класс → модель» не на чем строить.

**Решение на сейчас:** v1 — выбор сужает глобальный allow-list Admin, и довольно. Ось чувствительности (классификация ↔ допуск модели) и редакция перед отправкой — v2, при реальной комплаенс-потребности; дом контроля — рядом с `chat_models` / `model_assignments` в `ai-foundation/_workzone/data-model.html`, дом классификации — Knowledge Store.

## Web App — санитизация рендера ответа модели

**Суть:** ответ LLM рисуется в чате как markdown (ссылки, возможно картинки). Подложенная в найденный контент инструкция может заставить модель вывести ссылку / картинку вида `http://чужой/?leak=…`; чат, авто-загрузив ресурс, утащит контекст наружу — канал эксфильтрации (`query-engine/_workzone/protection.html#output`). Прорыв доступа закрыт в SQL + ACL, но рендер недоверенного вывода — нет.

**Контекст:** ответ рендерит оболочка Web App (тонкий клиент, `modules/web-app/index.html`) — модуль спроектирован, но топика рендера-санитизации в нём пока нет. CSP платформы (`auth-security/_workzone/protection.html`) покрывает статику, не вывод модели. Нужны: санитизация markdown, запрет авто-загрузки внешних ресурсов (`img` / `fetch`) из ответа, экранирование.

**Решение на сейчас:** принцип зафиксирован на стороне Query Engine (`protection.html#output`) как требование к рендеру; механизм спроектировать при дизайне оболочки Web App (чат, рендер сообщений). До тех пор — forward-зависимость, не v1-механизм Query Engine.
